# Акт-контроль: учёт оплат, проектов и закрывающих документов

Рабочий прототип дашборда для digital-агентства. Показывает цепочку

> проект / юрлицо → оплаты → этапы работ → статус закрывающих документов (актов)

и отвечает на главный операционный вопрос: сколько оплачено, какие проекты закрыты документами, где зависли акты и что требует внимания.

Тестовое задание сделано не как «таблица на Vue», а как небольшой управляемый бизнес-процесс с чётким разделением слоёв: данные, бизнес-логика, API, отображение.

---

## Стек и почему он

| Слой | Технология |
|------|------------|
| Backend | Laravel 12 (PHP 8.3) |
| Frontend | Vue 3 (SPA, Composition API) + Vue Router |
| API | REST (JSON), ресурсы Laravel |
| БД | SQLite (нулевая настройка; код не завязан на диалект, легко переключить на PostgreSQL/MySQL) |
| Сборка | Vite 7 + Tailwind CSS 4 |
| Тесты | PHPUnit (unit + feature) |

Стек выбран по заданию (Laravel + Vue + REST). SQLite взят ради «склонировал и запустил» без поднятия СУБД; вся работа с данными идёт через Eloquent, поэтому переезд на PostgreSQL это смена одной строки в `.env`.

## Сущности и связи

```
LegalEntity (юрлицо/клиент)
   │  1—N
   ├──> Project (проект)
   │       │  1—N
   │       └──> Payment (оплата) <──N—1── LegalEntity (плательщик)
   │                 │  1—1
   │                 └──> Act (закрывающий документ)
```

- **LegalEntity** — клиент/юрлицо: наименование, ИНН, КПП, ОГРН, расчётный счёт, банк, контактное лицо.
- **Project** — проект/направление работ, принадлежит клиенту (`legal_entity_id`).
- **Payment** — оплата. Привязана к проекту (`project_id`) и к плательщику (`legal_entity_id`). Несёт дату, сумму, назначение платежа, этап/тип услуги, номера счёта и договора. Плательщик вынесен отдельным полем намеренно: в реальности оплата по проекту может прийти от аффилированного юрлица, не совпадающего с владельцем проекта.
- **Act** — закрывающий документ. Один акт на оплату (`payment_id` уникален). Хранит флаги `is_sent` / `is_signed`, даты `sent_at` / `signed_at` и комментарий менеджера.

Схема в миграциях: `database/migrations/2025_06_01_0000*`.

## Где находится бизнес-логика

Логика вынесена из контроллеров. Контроллеры тонкие: разбирают запрос и отдают ресурс.

| Что | Где |
|-----|-----|
| Правило расчёта статуса акта (единственный источник правды) | `app/Enums/ActStatus.php` |
| Фильтрация, пагинация, расчёт сводки и агрегатов по проектам | `app/Services/PaymentService.php` |
| Разбор фильтров из запроса в типизированный объект | `app/Services/PaymentFilter.php` |
| Формат отдачи наружу | `app/Http/Resources/*` |
| Валидация изменения акта | `app/Http/Requests/UpdateActRequest.php` |

### Логика статусов

Статус акта **не хранится в БД, а вычисляется**, потому что правило «требует внимания» зависит от текущей даты и иначе устаревало бы. Весь расчёт собран в одном месте, `ActStatus::resolve()`:

- **Закрыт** — акт отправлен и подписан.
- **Не отправлен** — акт ещё не выставлен (в пределах нормы по сроку).
- **Ожидает подписи** — отправлен, но ещё не подписан (в пределах нормы).
- **Требует внимания** — оплата не закрыта документами слишком долго, а именно:
  - акт так и не отправлен спустя более 30 дней после оплаты, **или**
  - акт отправлен, но клиент не подписывает его более 14 дней.

Пороги вынесены в константы `ActStatus::ATTENTION_DAYS` и `ActStatus::SIGN_WAIT_DAYS`. Это уточнение базового правила из задания: «требует внимания» завязан не только на возраст оплаты, но и на то, на каком шаге документооборот застрял. Так статус точнее отражает операционную картину.

При изменении акта даты проставляются автоматически и поддерживается целостность: подпись неотправленного акта автоматически помечает его отправленным; снятие отправки снимает и подпись (нельзя считать подписанным неотправленный документ). См. `app/Http/Controllers/Api/ActController.php`.

## API

| Метод | Маршрут | Назначение |
|-------|---------|-----------|
| GET | `/api/dashboard/summary` | Сводка по текущим фильтрам |
| GET | `/api/payments` | Список оплат с пагинацией и статусом акта |
| GET | `/api/projects` | Агрегаты по проектам |
| GET | `/api/references` | Справочники для фильтров (проекты, юрлица, этапы, статусы) |
| PATCH | `/api/acts/{act}` | Отметить отправлен/подписан, задать комментарий |

Все списочные эндпоинты принимают одинаковый набор фильтров: `project_id`, `legal_entity_id`, `date_from`, `date_to`, `service_stage`, `act_status`, `search` (по назначению платежа, названию проекта и клиента). Сводка и таблица грузятся с одними фильтрами, поэтому цифры в карточках всегда совпадают с содержимым таблицы.

## Фронтенд

SPA на Vue 3, точка входа `resources/js/app.js`. Доступ к API изолирован в одном модуле `resources/js/api.js`, компоненты про axios не знают.

```
resources/js/
  App.vue              layout и навигация
  router.js            маршруты (Оплаты / Проекты)
  api.js               единый клиент API
  format.js            форматирование денег и дат
  views/
    DashboardView.vue  оркестрация: сводка + фильтры + таблица + пагинация
    ProjectsView.vue   таблица проектов и документооборота
  components/
    SummaryCards.vue   карточки сводки
    FiltersBar.vue     панель фильтров
    PaymentsTable.vue  таблица оплат, сортировка, переключатели актов
    Pagination.vue     постраничная навигация
    StatusBadge.vue    бейдж статуса
```

Изменение акта в таблице сразу уходит на сервер (`PATCH`), строка и сводка обновляются на месте, без полной перезагрузки. Изменения статусов сохраняются.

## Допущения

- Авторизация и роли не реализованы (вне рамок задания).
- На каждую оплату приходится ровно один акт. Если акта нет, оплата трактуется как «акт не отправлен».
- Мок-данные сгенерированы (детерминированно, фиксированный seed) и имитируют банковскую выписку: 6 юрлиц, 10 проектов, 30 оплат, акты во всех четырёх статусах. Реальные финансовые данные не используются.
- Плательщик в сидере совпадает с владельцем проекта, но модель поддерживает их различие.

## Запуск локально

Нужны PHP 8.2+, Composer, Node 18+.

```bash
composer install
npm install

cp .env.example .env
php artisan key:generate

php artisan migrate --seed   # создаст и наполнит БД (SQLite)

npm run build                # собрать фронтенд
php artisan serve            # http://127.0.0.1:8000
```

Для разработки с hot-reload вместо `npm run build` запустите `npm run dev` в отдельном терминале параллельно с `php artisan serve`.

## Деплой

Проект это монолит Laravel: фронтенд собирается в `public/build` и отдаётся тем же приложением, отдельный хостинг для SPA не нужен.

- **Railway / Render** подойдут из коробки: PHP-окружение, команда сборки `composer install --no-dev && npm install && npm run build`, старт через `php artisan serve` или встроенный веб-сервер. SQLite работает на постоянном диске; для продакшена лучше PostgreSQL (сменить `DB_CONNECTION`).
- **Docker.** В корне есть `Dockerfile`, собирающий ассеты и поднимающий приложение одним контейнером:

  ```bash
  docker build -t acts-dashboard .
  docker run -p 8000:8000 acts-dashboard
  ```

Vercel для Laravel не подходит (это платформа под фронтенд и serverless-функции), поэтому для бэкенда выбран Railway/Render или Docker.

## Куда это масштабируется

- **Фильтр по статусу акта** сейчас применяется в PHP-слое (после SQL-фильтров), чтобы правило статуса жило в одном месте, `ActStatus`. На текущем объёме это дёшево. На больших данных правило выносится в денормализованную колонку `status`, обновляемую по расписанию или через очередь при изменении акта; единый источник правды (`ActStatus`) при этом сохраняется, меняется только место применения.
- **Импорт из банковской выписки.** Сейчас данные приходят из сидера. В бою между выпиской и таблицей `payments` встаёт слой импорта: парсер (PDF/CSV) → нормализация → сопоставление платежа с проектом по номеру договора/назначению → создание `Payment` и пустого `Act`. Контроллеры и фронтенд при этом не меняются, добавляется один Importer.
- **Очереди и вебхуки** (уведомления менеджеру о просроченных актах, автогенерация документов) подключаются поверх существующих сущностей без переделки модели.
